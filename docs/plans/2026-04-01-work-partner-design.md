# Work Partner 设计文档

日期：2026-04-01

## 1. 项目目标

Work Partner 是一个将学习任务系统与电子宠物成长系统结合的产品。用户可以围绕一个学习周期设定目标，自主发布任务、给他人发布任务，并通过完成任务驱动电子宠物成长、进化和外观变化。

本项目第一阶段的核心目标不是做一个复杂的多人协作平台，而是验证以下产品闭环：

1. 用户能围绕学习目标持续获得任务推进感。
2. 用户能通过任务完成获得即时经验和成长反馈。
3. 用户能通过电子宠物进化、装扮和愿望机制获得长期成就感。
4. 用户能通过视频讲解获得额外奖励，把费曼学习法纳入产品循环。

## 2. 核心产品定位

产品定位为小组协作型学习成长系统，但协作是辅助能力，核心价值仍然是个人成长可视化。

产品体验重心如下：

1. 强化成长进化的成就感。
2. 用电子宠物承载学习成果，而不是只做任务管理。
3. 任务完成后立即给正反馈，不用审批驱动主成长。
4. 通过视频讲解、许愿外观和分支进化拉高长期留存。

## 3. 目标用户与年龄适配

产品需要覆盖 3 岁到 80 岁的广泛年龄段，因此必须将玩法层与视觉层拆开。

统一规则层：

1. 所有用户共用同一套任务、经验、进化、愿望和反馈机制。
2. 不同年龄段不改变核心玩法规则。

可配置视觉层：

1. 用户初始化时先选择宠物风格，再选择具体宠物。
2. 宠物风格独立成素材包，不影响数值和成长逻辑。

推荐的风格包如下：

1. 童趣低龄风：更圆润、颜色更亮、表情更直接。
2. 经典宠物风：中性、通用、适配范围广。
3. 拟人伙伴风：更像陪伴角色，适合青少年和成人。
4. 二次元角色风：强调角色感、成长立绘和收藏感。

## 4. 核心产品循环

系统以学习周期目标作为最高层驱动。

核心循环如下：

1. 用户创建一个学习周期目标。
2. 系统围绕目标生成知识点任务。
3. 用户补充临时任务，或由小组成员互相发布任务。
4. 用户完成任务后立即获得经验。
5. 宠物根据经验升级并进入后续进化阶段。
6. 用户可上传费曼式讲解视频，获得特殊道具。
7. 用户通过许愿机制和外观奖励影响宠物后续进化表现。

这个循环中，经验负责即时成长反馈，视频反馈负责高价值学习行为奖励，许愿机制负责长期外观驱动。

## 5. 任务系统设计

### 5.1 任务主线

任务分成两条主线：

1. 临时任务：用于承接短期、即时安排和协作指派任务。
2. 知识点任务：围绕学习目标自动生成，用于推进真正的学习主线。

### 5.2 目标驱动生成

每个学习周期先由用户创建一个目标，例如“10 天掌握 Python 基础语法”或“本周掌握牛顿第二定律”。

系统基于目标先生成一版知识点路径，再根据进度持续补发任务，而不是一次性生成全部内容。

任务生成采用以下策略：

1. 初次拆解：将目标拆成一组知识点任务。
2. 进度跟踪：根据已完成情况决定下一个知识点任务。
3. 补偿生成：当某个知识点推进缓慢时，生成更小颗粒的辅助任务。

### 5.3 经验规则

经验发放保持简单直接：

1. 任务完成立即发放经验。
2. 知识点任务的经验值略高于临时任务。
3. 同一目标下连续完成任务时可给连击奖励。

系统不要求审批后才发主经验，主循环必须足够轻量。

## 6. 协作机制

协作机制控制在轻量范围内，避免前期复杂化。

MVP 协作动作只保留：

1. 给自己发布任务。
2. 给别人发布任务。
3. 查看成员成长状态。

别人发给你的任务仍然进入你的任务池，可以归类为临时任务或挂到某个知识点目标下。完成后直接给经验，不引入复杂验收。

协作展示更适合采用成长墙而不是强排名榜单，建议展示：

1. 宠物当前阶段。
2. 最近成长记录。
3. 已解锁外观元素。
4. 费曼讲解徽章或特殊道具。

## 7. 宠物系统设计

### 7.1 初始化选择

用户首次进入系统时需要先选宠物风格，再选具体宠物。

MVP 建议至少提供 4 个初始宠物成长原型，每个原型都需要有独立初始化图片资源。

推荐成长原型如下：

1. 专注型
2. 探索型
3. 表达型
4. 协作型

每个原型可以被映射到多个风格包中，保证玩法统一、视觉可扩展。

### 7.2 资源要求

每种风格包建议优先准备：

1. 风格封面图
2. 初始宠物图
3. 一阶进化图

后续逐步补充高阶进化图、特殊分支图和装扮素材。

### 7.3 进化结构

宠物进化分两层：

1. 等级进化：由经验值驱动，体现“做任务就成长”。
2. 分支进化：由行为偏好、特殊道具和外观融合共同驱动。

等级进化建议采用：

1. 幼体
2. 成长期
3. 进阶体
4. 高阶体

系统不设计宠物退化。最多通过疲惫或待鼓励状态提示用户近期节奏偏慢，但不回退等级，保证正反馈体验。

## 8. 费曼学习法反馈机制

视频讲解作为高价值反馈机制，不作为主线通关门槛。

规则如下：

1. 用户完成某个知识点任务后，可选择录制简短讲解视频。
2. 视频内容强调“用自己的话讲给别人听”，体现费曼学习法。
3. 上传后标记该知识点为已讲解。
4. 用户获得特殊道具或讲解徽章。

推荐特殊道具示例：

1. 讲解徽章
2. 灵感结晶
3. 记忆羽片

这些道具主要用于触发稀有分支进化、装扮解锁或成长标记，不替代主经验系统。

## 9. 外观愿望机制

### 9.1 机制目标

外观愿望机制是经验成长之外的第二条长期驱动力。用户不是只为了升级而完成任务，也是在为宠物积累理想外观。

### 9.2 触发规则

愿望机制不是绑定某个指定任务，而是一个周期内的成长事件。

触发条件如下：

1. 某个学习周期中未完成任务数必须大于 3。
2. 满足条件后，系统触发一次许愿事件，允许用户许愿。

### 9.3 许愿流程

许愿事件触发后，流程如下：

1. 用户从外观库选择一个想要的元素，加入当前愿望。
2. 系统从该周期内当前未完成任务中随机抽取一个任务。
3. 该任务成为本次愿望绑定任务。
4. 当绑定任务被完成时，系统发放对应愿望奖励。

### 9.4 愿望奖励与进化融合

愿望奖励不会在获得时强制立即穿戴，而是进入待融合状态。

后续在宠物下一次进化时，用户可以选择：

1. 将该愿望外观融合到本次进化外观中。
2. 暂不融合，保留奖励，未来进化时再使用。

如果用户本次不使用该奖励，奖励必须永久保留，不失效。

### 9.5 外观模块建议

为了控制实现复杂度，MVP 外观元素优先采用模块化部件，而不是整只宠物完全重绘。

推荐部件层级：

1. 头部饰品
2. 身体装饰
3. 手持物
4. 表情特效
5. 背景挂件

## 10. 页面结构设计

MVP 建议先实现 5 个核心页面：

1. 首页 / 成长看板：当前宠物、等级、经验、周期目标、今日任务、许愿状态。
2. 任务页：按临时任务、知识点任务、他人指派任务分栏展示。
3. 目标页：创建和查看学习周期目标、知识点路径和进度。
4. 宠物页：风格选择、进化树、外观管理、心愿单和待融合奖励。
5. 反馈页：视频上传、讲解记录、特殊道具和知识点反馈状态。

## 11. 数据模型

MVP 数据结构建议如下：

1. `User`：用户基础信息。
2. `PetStylePack`：风格包信息，如低龄风、经典风、拟人风、二次元风。
3. `PetTemplate`：初始宠物模板信息。
4. `Pet`：用户当前宠物实例，包含等级、经验、当前阶段、当前外观组合。
5. `GoalCycle`：学习周期目标。
6. `KnowledgeNode`：知识点节点。
7. `Task`：任务实体，包含类型、来源、所属周期、状态、经验值、绑定愿望标记。
8. `Wish`：一次愿望事件，包含选择的外观元素、触发时间、绑定任务、发放状态。
9. `AppearanceItem`：可解锁外观元素。
10. `PendingAppearanceReward`：已获得但尚未融合进化的外观奖励。
11. `VideoFeedback`：视频讲解记录。
12. `SpecialItem`：通过视频讲解获得的特殊道具。

## 12. 存储方案

第一阶段不接数据库，主存储采用浏览器本地存储。

推荐方案如下：

1. 主存储采用浏览器 `IndexedDB`。
2. 用户可将本地存档导出为 `work-partner-save.json`。
3. 用户可通过导入 JSON 文件恢复本地进度。

这样设计的原因如下：

1. MVP 能最快形成可玩的浏览器原型。
2. 数据结构较多，使用 `IndexedDB` 比 `localStorage` 更稳妥。
3. 导入导出机制可作为备份和迁移方案。

第一阶段不做：

1. 云同步
2. 实时多人共享
3. 服务端数据库

小组协作在 MVP 中采用本地模拟数据或本地多成员状态即可。

## 13. MVP 范围

第一阶段必须具备以下闭环能力：

1. 创建学习周期目标。
2. 根据目标生成知识点任务。
3. 用户创建临时任务。
4. 用户给别人创建任务。
5. 完成任务立即获得经验。
6. 宠物根据经验升级和进化。
7. 初始化时可选择不同风格和多个初始宠物图片。
8. 支持视频讲解上传记录和特殊道具奖励。
9. 支持外观心愿单。
10. 当未完成任务数大于 3 时触发许愿事件。
11. 系统从未完成任务中随机绑定一个任务到愿望。
12. 完成绑定任务后发放愿望奖励。
13. 下次进化时可选择是否融合愿望外观。
14. 未使用的愿望奖励永久保留。
15. 数据使用 IndexedDB 本地保存，并支持 JSON 导入导出。

## 14. 后续扩展方向

在 MVP 验证后，可扩展以下方向：

1. 云同步与账号体系
2. 真正的小组协作与跨设备共享
3. 更复杂的知识点拆解与智能任务生成
4. 更丰富的宠物进化树
5. 稀有外观、成就图鉴和收藏系统
6. 视频反馈内容分析与学习质量评估

## 15. 设计结论

Work Partner 的第一阶段应当优先验证“学习目标推进感 + 宠物成长成就感 + 愿望外观期待感”的三重闭环。

在实现策略上，第一阶段不引入数据库，采用浏览器 IndexedDB 作为主存储，并用 JSON 导入导出补足备份和迁移能力。这样既能快速形成可玩的原型，也为后续演进到完整协作系统保留空间。
