# Work Partner 补充需求设计

日期：2026-04-01

## 1. 文档目标

本文档用于补充当前 MVP 之后最需要统一的需求边界，重点解决以下问题：

- 当前 5 个页面大多还是页面内局部状态，缺少统一的应用级状态源。
- 目标、任务、宠物、反馈、愿望、奖励之间还没有形成完整的数据闭环。
- 当前界面和测试已经补齐中文文案，但示例数据、持久化数据和演示数据需要持续保持同一套产品语言。
- 后续继续开发时，需要更明确的验收口径，避免“测试通过但产品链路没打通”。

## 2. 当前实现现状

当前仓库已经具备以下能力：

- React + TypeScript + Vite 单页应用壳
- 5 个核心页面：成长看板、任务、目标、宠物、反馈
- Dexie + IndexedDB 本地存储仓库层
- JSON 导入导出
- 演示种子数据
- 基础单元测试与页面级测试

当前最主要的实现缺口：

- `GoalsPage`、`TasksPage`、`PetPage`、`FeedbackPage` 仍以页面本地 `useState` 为主
- 目标创建不会真实写入统一快照，也不会驱动任务页变化
- 任务完成不会真实影响宠物、愿望、奖励、反馈记录
- 反馈记录不会沉淀到统一存档
- 演示数据和运行时状态虽然都存在，但仍缺少真正的单一事实来源

## 3. 补充需求范围

### 3.1 统一应用快照

后续实现应以 `AppSnapshot` 作为单一事实来源，页面只读取和修改统一快照，不再各自维护独立业务状态。

要求：

- 页面初始化时从仓库层加载快照
- 页面操作统一落到快照更新函数
- 持久化写入由统一应用层负责，而不是散落在各页面
- 导入导出必须基于同一份快照

### 3.2 目标与任务联动

目标创建后，系统必须生成对应知识任务，并且这些任务在任务页可见。

要求：

- 创建目标时立即生成一组知识任务
- 任务记录必须带有 `goalId`
- 任务页至少支持按“全部任务 / 某目标任务”展示
- 删除或结束目标时，应定义任务处理策略

MVP 补充策略：

- 先不做目标删除
- 先不做复杂筛选
- 先保证“创建目标后任务出现”这条主链路成立

### 3.3 任务完成与宠物成长联动

任务完成必须驱动宠物经验变化，并影响愿望机制和进化流程。

要求：

- 完成任务后任务状态更新为已完成
- 宠物经验增加
- 当经验满足阈值时触发进化判定
- 愿望事件依赖未完成任务数进行判定

MVP 补充策略：

- 经验公式仍保持简单
- 不引入复杂衰减或多种奖励倍率
- 先打通“任务完成 -> 宠物成长 -> 愿望状态更新”

### 3.4 愿望与外观奖励联动

愿望不是单独的静态文本，而是一个真实的快照数据对象。

要求：

- 愿望事件要写入 `wishes`
- 愿望绑定的任务必须可追溯
- 完成绑定任务后，外观奖励进入 `pendingAppearanceRewards`
- 未融合奖励必须持久保留

### 3.5 反馈与特殊奖励联动

反馈记录必须成为可持久化的业务数据，而不是页面临时数组。

要求：

- 新增反馈记录写入 `videoFeedbacks`
- 奖励写入 `specialItems`
- 反馈页可回看历史记录
- 导出导入后反馈记录和奖励应被完整恢复

### 3.6 中文化约束

后续所有新增用户可见内容默认使用中文。

要求：

- 页面文案、表单标签、状态文案、按钮、提示语使用中文
- 默认示例内容、演示种子数据、奖励展示文本使用中文
- 内部代码标识、路由路径、类型名可以继续保留英文

## 4. 关键验收标准

### 4.1 产品链路验收

必须满足以下链路：

1. 创建目标后，任务页出现对应知识任务
2. 完成任务后，任务状态变更且宠物经验增加
3. 当未完成任务数满足条件时，可以触发愿望事件
4. 完成绑定任务后，出现待融合外观奖励
5. 提交反馈后，可看到反馈记录和特殊奖励
6. 导出后清空本地数据，再导入，可以恢复上述状态

### 4.2 工程验收

必须满足以下工程条件：

- `npm test` 通过
- `npm run build` 通过
- 核心链路至少有一层自动化测试覆盖
- README 和工程文档与实际命令一致

## 5. 推荐实现顺序

建议按以下顺序推进：

1. 建立统一应用状态容器
2. 打通目标创建与任务生成
3. 打通任务完成与宠物成长
4. 打通愿望与奖励持久化
5. 打通反馈与特殊奖励持久化
6. 补全导入导出回放测试

## 6. 非目标

以下内容暂不纳入当前补充需求：

- 云同步
- 多账号
- 真正的多人协作
- 服务端 API
- 权限系统
- 音视频上传到远端对象存储
- 复杂推荐算法

## 7. 结论

当前 MVP 已经具备“可运行、可测试、可演示”的基础，但下一阶段的核心不是继续堆页面，而是统一状态源并打通业务链路。

后续所有新增功能都应优先服从这条主线：

- 一个快照
- 一套持久化
- 一条完整产品闭环
